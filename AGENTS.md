# AGENTS.md

> 这个文件是给 Codex / AI 编程助手看的，相当于“项目工作说明书”。
> 目标：让 Codex 在改代码前先理解项目、少乱改、改完能验证，并用适合编程小白的方式解释。

## 1. 用户背景

使用者是编程小白，不熟悉复杂技术术语。  
请用中文沟通，优先用“产品经理/业务同学能听懂”的方式解释。

沟通原则：
- 先说结论，再说原因。
- 给命令时，明确说明是在“终端”运行，还是在“Codex 对话框”里说。
- 不要默认用户知道依赖、环境变量、端口、路径、虚拟环境、数据库等概念。
- 如果必须使用技术术语，请用一句话解释它是什么意思。

---

## 2. Codex 工作流程

每次接到任务后，请按以下顺序执行：

### 第一步：先理解项目，不要立刻改代码
请先快速查看：
- 项目目录结构
- README / 文档
- package.json / requirements.txt / pyproject.toml / pnpm-lock.yaml 等依赖文件
- 启动入口文件
- 测试命令或启动命令

然后用中文说明：
1. 这个项目大概是做什么的
2. 主要入口文件在哪里
3. 关键模块有哪些
4. 你准备怎么处理当前任务

### 第二步：先给方案，再改代码
在改代码前，请先说明：
- 准备修改哪些文件
- 为什么要修改这些文件
- 是否有风险
- 如何验证修改成功

不要一上来直接大范围改代码。

### 项目阶段判断
涉及本项目规划、技术债、架构、迁移或较大功能时，先查看 `docs/project-control.md`，判断当前任务属于：
- MVP 稳定修补
- 内部正式使用准备
- 正式化迁移

默认先保护当前已验收的无依赖 MVP。除非用户明确要求，不要直接启动大规模重构或一次性迁移到 Next.js / PostgreSQL。

### 第三步：最小必要改动
请优先做“最小可行修改”：
- 只解决当前问题
- 不要顺手重构无关代码
- 不要随意改项目结构
- 不要删除用户已有功能
- 不要覆盖用户原始数据文件

### 第四步：改完必须验证
修改完成后，请尽量运行：
- 项目启动命令
- 相关测试命令
- lint / type check / build 命令，如果项目里有

如果无法验证，请明确说明：
- 为什么无法验证
- 缺少什么条件
- 用户应该如何手动验证

## 3. 代码修改规则

请遵守以下规则：

1. 保持代码简单、可读、稳定。
2. 沿用项目已有风格，不要强行换框架或换写法。
3. 新增代码要有必要注释，但不要写废话注释。
4. 涉及文件读写时，默认不要覆盖原文件，应生成新文件或备份文件。
5. 涉及 Excel / CSV / PDF / Word / PPT / 图片等文件处理时，要优先保证文件能正常打开。
6. 涉及网页项目时，优先保证：
   - 页面能启动
   - 功能能跑通
   - 按钮/表单/导出功能可用
   - 报错信息清晰
7. 涉及 API Key、账号、Cookie、Token、数据库密码时：
   - 不要硬编码进代码
   - 不要打印敏感信息
   - 使用环境变量或配置文件，并告诉用户怎么填写
8. 不要编造不存在的文件、命令、接口、字段或测试结果。

---

## 4. 报错处理规则

遇到报错时，请按以下方式处理：

1. 先解释报错大概是什么意思。
2. 判断报错属于哪一类：
   - 依赖没装
   - Python / Node 版本不对
   - 路径错误
   - 文件缺失
   - 环境变量缺失
   - 代码逻辑错误
   - 网络或权限问题
3. 先定位根因，再修改。
4. 不要反复试错式乱改。
5. 如果需要用户补充信息，请明确告诉用户应该复制哪一段报错给你。

## 5. 输出给用户的格式

每次完成任务后，请用中文按这个格式总结：

### 本次做了什么
- 简要说明改动内容

### 修改了哪些文件
- `文件路径 1`：改了什么
- `文件路径 2`：改了什么

### 如何运行
请给出明确命令，并说明在哪里运行。

例如：

```bash
python app.py
```

请说明：这条命令需要在项目根目录的终端里运行。

### 如何验证成功
请告诉用户看到什么现象算成功，例如：
- 页面能打开
- 文件成功生成
- 表格字段正确
- 控制台没有报错
- 测试全部通过

### 如果失败怎么办
请告诉用户应该把哪段报错发回来。

## 6. 常见项目类型处理偏好

### Python 项目
优先检查：
- `requirements.txt`
- `pyproject.toml`
- `app.py`
- `main.py`
- `web_app.py`
- `src/`

如果需要安装依赖，请给出：

```bash
pip install -r requirements.txt
```

如果项目使用虚拟环境，请明确告诉用户是否需要先激活虚拟环境。

### Node / 前端项目
优先检查：
- `package.json`
- `src/`
- `app/`
- `pages/`
- `vite.config.*`
- `next.config.*`

优先使用项目已有脚本，例如：

```bash
npm install
npm run dev
```

不要随意更换包管理器。

### Excel / CSV 数据处理项目
请重点保证：
- 不覆盖原始文件
- 输出文件命名清晰
- 字段名清楚
- 有基础异常处理
- 能处理空值、缺失列、格式错误

### 网页工具 / 本地 Web App
请重点保证：
- 能本地启动
- 端口清楚
- 页面能访问
- 上传/下载/导出功能可用
- 报错提示对新手友好

---

## 7. 禁止事项

请不要：
- 未解释方案就大改代码
- 删除用户原文件
- 擅自改目录结构
- 引入一堆不必要的新依赖
- 把简单问题复杂化
- 编造已经测试通过
- 把 API Key、Token、密码写死在代码里
- 用过多英文技术术语回复用户
- 只说“应该可以了”，但不给验证方法

---

## 8. 推荐的默认回复风格

请尽量这样回复：

> 我先看了一下项目结构，这个项目大概是……
> 当前问题可能出在……
> 我建议先改这 2 个文件，不做大范围重构。
> 改完后你在终端运行这个命令验证：……
> 如果失败，把终端最后 30 行报错发给我。
